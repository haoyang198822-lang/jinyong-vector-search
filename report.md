# 向量数据库学习打卡 - 学习报告

## 一、项目目标

本项目旨在通过《倚天屠龙记》原文构建一个完整的向量数据库检索系统，实现：
- 文本向量化：将小说文本转换为高维向量表示
- 高效检索：使用 Faiss 向量索引实现毫秒级相似度搜索
- 语义匹配：支持自然语言查询，返回相关内容片段
- 交互式应用：提供 Streamlit Web 界面方便用户使用

## 二、技术方案

### 2.1 技术栈

| 组件 | 技术选型 | 说明 |
|------|----------|------|
| 向量模型 | text-embedding-v4 | 阿里云通义千问嵌入模型，1024维输出 |
| 向量接口 | OpenAI 兼容 API | 通过 DashScope 实现，base_url 配置 |
| 向量索引 | Faiss IndexFlatIP | 内积索引，支持归一化向量的余弦相似度 |
| 元数据存储 | JSON | 包含章节、行号、位置等丰富信息 |
| Web 框架 | Streamlit | 快速构建数据交互应用 |
| 数据源 | 《倚天屠龙记》精校版 TXT | GBK 编码，约 2MB |

## 三、实现流程

### 3.1 文本预处理

1. **编码处理**：使用 GBK 编码读取 TXT 文件，忽略无法解码的字符
2. **章节检测**：通过正则表达式匹配中文数字章节标题
   - 支持「第X章」「第X节」「第X回」等多种格式
   - 将中文数字（一二三四...）转换为阿拉伯数字
   - 去重处理，保留最早出现的章节

### 3.2 文本分块策略

采用**句号边界优化**的分块方式：

```
原始文本: "...内容...。下一段内容...再一段内容...。"

如果当前块+新行 > 400字符:
    ├── 查找最后一个句号「。」
    ├── 若句号位置 > 60%: 在句号处断开
    └── 否则: 强制断开

重叠部分: 保留最后100字符到新块开头
```

**优势**：
- 保证语义完整性（以句子为单位）
- 保留上下文信息（重叠设计）
- 避免截断关键词

### 3.3 向量化流程

```python
# API 调用参数
{
    "model": "text-embedding-v4",
    "base_url": "https://dashscope.aliyuncs.com/compatible-mode/v1",
    "batch_size": 10,
    "interval": 0.1s  # 避免限流
}
```

- 批量提交10个文本块
- 每次请求间隔0.1秒
- 获取1024维浮点向量

### 3.4 Faiss 索引构建

```python
# 索引类型: IndexFlatIP (内积索引)
# 特点: 
#   - 无需训练，可直接添加向量
#   - 精确搜索，无精度损失
#   - 适合百万级以下数据

# 向量归一化
normalized = embeddings / norms  # L2归一化
index.add(normalized)

# 搜索时使用内积 = 余弦相似度（归一化后）
```

## 四、遇到的问题和解决方案

### 4.1 编码问题

**问题**：文本文件为 GBK 编码，直接使用 UTF-8 读取会报错

**解决**：
```python
with open(file_path, 'r', encoding='gbk', errors='ignore') as f:
    text = f.read()
```

### 4.2 章节识别不完整

**问题**：正则表达式只匹配到部分章节

**解决**：
- 增加多种章节格式支持（章/节/回）
- 优化中文数字转换逻辑
- 添加去重机制

### 4.3 API 限流

**问题**：连续快速调用 API 导致限流

**解决**：
```python
for i in range(0, len(texts), BATCH_SIZE):
    # 处理批次
    time.sleep(API_INTERVAL)  # 每次间隔0.1秒
```

### 4.4 内存占用

**问题**：大文本向量化后内存占用较高

**解决**：
- 分批处理，控制内存使用
- 使用 numpy float32 节省空间
- 及时释放中间变量

## 五、TF-IDF vs text-embedding-v4 对比分析

### 5.1 原理对比

| 维度 | TF-IDF | text-embedding-v4 |
|------|--------|-------------------|
| **原理** | 词频-逆文档频率 | 深度学习语义编码 |
| **向量维度** | 词汇表大小 (万级) | 固定1024维 |
| **语义理解** | 仅词汇匹配 | 上下文语义 |
| **训练数据** | 无需训练 | 基于海量文本预训练 |
| **同义词处理** | ❌ 无法识别 | ✅ 可理解 |

### 5.2 检索效果对比

| 查询 | TF-IDF 结果 | text-embedding-v4 结果 |
|------|-------------|------------------------|
| "张无忌学武功" | 可能返回无关的战斗描写 | 返回学习/修炼相关段落 |
| "赵敏的性格" | 词汇匹配度低 | 返回性格描写、互动场景 |
| "倚天剑下落" | 精确词匹配 | 返回相关剧情线索 |

### 5.3 优劣势总结

**TF-IDF 优势**：
- 无需 API 调用，离线可用
- 计算简单，速度快
- 可解释性强

**TF-IDF 劣势**：
- 无法处理语义相关性
- 维度过高（词汇表膨胀）
- 稀疏表示，存储效率低

**text-embedding-v4 优势**：
- 语义理解能力强
- 固定维度，便于索引
- 预训练模型，效果好

**text-embedding-v4 劣势**：
- 需要 API 调用（有成本）
- 模型不可控
- 依赖网络连接

## 七、改进方向

### 7.1 混合检索

将向量检索与关键词检索结合：
```
Final Score = α × VectorScore + (1-α) × BM25Score
```

**优势**：
- 既考虑语义相关性
- 又保证关键词精确匹配

### 7.2 Reranker 模型

使用交叉编码器对初筛结果重排序：
```
Query + Document → Relevance Score
```

**效果**：可将 Top-K 准确率提升 10-20%

### 7.3 章节级别索引

当前以固定长度分块，可考虑：
- 以章节为单位建立索引
- 章节内部分块，层级检索
